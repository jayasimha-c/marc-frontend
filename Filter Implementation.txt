Filter Flow Analysis: Source Frontend → Backend

  1. Source Frontend (PrimeNG/AG-Grid Legacy Pattern)

  Model: PaginationModel = { first, rows, sortOrder, sortField, filters: object|null, page?, globalFilter? }

  Default: { first: 1, rows: 10, sortOrder: 0, sortField: "", filters: {}, page: 0 }

  Component builds the event like:
  {
    first: event.first,
    rows: event.rows,
    sortOrder: event.sortOrder,
    sortField: event.sortField || "",
    filters: event.filters || null,
    page: Math.round(event.first / 10) + 1,
    globalFilter: event.globalFilter || null
  }

  Service sends as GET query params:
  this.http.get(`endpoint?first=${e.first}&rows=${e.rows}&sortOrder=${e.sortOrder}&sortField=${e.sortField}&filters=${encodeURI(JSON.stringify(e.filters))}&global
  Filter=${e.globalFilter}`)

  Filters are a PrimeNG-style object like {"bname":[{"value":"x","matchMode":"contains"}]} — JSON-stringified and URI-encoded as a query parameter.

  ---
  2. Backend — Two Distinct Patterns

  Pattern A: Legacy GET Endpoints (adhocAnalysis, analysis, simulations)

  - Accept filters as a JSON-stringified query parameter in PrimeNG format
  - Backend parses the JSON string back into a filter object
  - Used by: analysis/adhocRiskViolaitonGrid, analysis/simulationResultSummary, etc.

  Pattern B: New POST Endpoints (ServiceNow)

  - Accept @RequestBody GridFilterRequest as POST body
  - GridFilterRequest: { page, size, sortField, sortDirection (ASC/DESC enum), filters: List<ColumnFilter>, globalFilter }
  - ColumnFilter: { field, operator (CONTAINS/EQUALS/STARTS_WITH/ENDS_WITH/IN/etc.), value, values }
  - sortDirection is a Java enum — must be uppercase "ASC" or "DESC"
  - filters must be an array of ColumnFilter, not an object
  - Used by: servicenow/sync/data/{id}/users/filtered, roles/filtered, groups/filtered

  ---
  3. Target Frontend (app-advanced-table)

  The target's advanced-table emits TableQueryParams:
  { pageIndex (1-based), pageSize, sort: {field, direction}, filters: {fieldName: 'value'}, globalSearch }

  Conversion needed per endpoint type:
  ┌────────────┬───────────────────────────────┬──────────────────────────────────┬────────────────────────────────────┐
  │   Field    │     advanced-table emits      │      Pattern A (GET legacy)      │ Pattern B (POST GridFilterRequest) │
  ├────────────┼───────────────────────────────┼──────────────────────────────────┼────────────────────────────────────┤
  │ page       │ pageIndex (1-based)           │ first = (pageIndex-1)*pageSize+1 │ page = pageIndex - 1 (0-based)     │
  ├────────────┼───────────────────────────────┼──────────────────────────────────┼────────────────────────────────────┤
  │ size       │ pageSize                      │ rows = pageSize                  │ size = pageSize                    │
  ├────────────┼───────────────────────────────┼──────────────────────────────────┼────────────────────────────────────┤
  │ sort field │ sort.field                    │ sortField                        │ sortField                          │
  ├────────────┼───────────────────────────────┼──────────────────────────────────┼────────────────────────────────────┤
  │ sort dir   │ sort.direction ('asc'/'desc') │ sortOrder (1 or -1)              │ sortDirection ('ASC'/'DESC')       │
  ├────────────┼───────────────────────────────┼──────────────────────────────────┼────────────────────────────────────┤
  │ filters    │ { field: 'value' } object     │ JSON-stringified PrimeNG object  │ ColumnFilter[] array               │
  ├────────────┼───────────────────────────────┼──────────────────────────────────┼────────────────────────────────────┤
  │ search     │ globalSearch                  │ globalFilter                     │ globalFilter                       │
  └────────────┴───────────────────────────────┴──────────────────────────────────┴────────────────────────────────────┘

> is appraoch A cause any performance issues ?

● No. Approach A is purely a code organization change — it moves the same conversion logic from 5+ scattered copies into one shared function.

  At runtime:
  - Same HTTP requests go to the same endpoints
  - Same payload format, same response format
  - The utility just maps a few fields (subtract 1 from pageIndex, uppercase a string, loop through a small filters object) — nanoseconds of work
  - No extra network calls, no middleware, no serialization overhead

  The only "difference" is the legacy GET endpoints send filters as a JSON-stringified query param (?filters=...), which is slightly less efficient than a POST
  body for large filter sets — but that's already the case today. Approach A doesn't change that.

  TL;DR: Zero performance impact. It's a refactor, not an architectural change.
